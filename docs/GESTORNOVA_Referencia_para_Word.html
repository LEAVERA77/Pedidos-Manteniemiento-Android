0. API, Backend y Frontend — analogía del restaurante

0.1. ¿Qué significa API?

API son las siglas de Application Programming Interface (Interfaz de Programación de Aplicaciones).

En lenguaje sencillo: es un traductor o mensajero.

Analogía del restaurante: Vos sos el cliente (la app de Android o la web). La cocina es donde se prepara la comida (la base de datos Neon). La API es el mesero. No entrás a la cocina a cocinar: la app no debería acceder directo a la base por seguridad y orden. Le pedís al mesero (API) lo que necesitás; el mesero lleva el pedido a la cocina, espera y te trae el plato (los datos en formato JSON u otro acordado).

0.2. ¿Qué es el Backend?

El Backend es todo lo que pasa "detrás de escena" en el servidor: el usuario no lo ve. Es el cerebro que valida reglas, permisos y se conecta a la base de datos.

En GestorNova, el backend (código en la carpeta api/ del proyecto, ejecutado en el servidor) se encarga de:

  • Seguridad: comprobar que quien pide datos tenga permiso (por ejemplo con JWT).
  • Lógica: reglas de negocio, subida de archivos, integraciones (por ejemplo Cloudinary según la API).
  • Conexión: hablar con Neon para guardar o leer información sin exponer la "llave maestra" de la base en el teléfono o en una web pública.

0.3. ¿Para qué sirve en GestorNova?

La aplicación Android (y la interfaz web) es el Frontend — la cara visible. Para muchas funciones necesita hablar con el Backend a través de la API (pedidos HTTP a URLs concretas).

Ejemplo — actualización de la app (OTA):

  1. El Frontend (la app) pregunta: "¿Hay una versión nueva?"
  2. Llama a la API en una dirección como: https://nexxo-api-418k.onrender.com/api/app-version (definida en app/src/main/assets/app_update_config.json en el APK).
  3. El Backend en Render recibe el pedido, consulta la base Neon, lee la tabla app_version y obtiene, por ejemplo, la versión 10 y el enlace del APK en Drive.
  4. La API responde a la app con JSON: versión, URL de descarga, notas, etc.
  5. El Frontend muestra el aviso de actualización al usuario.

¿Por qué importa?

  • Centralización: si cambiás el enlace del APK, actualizás la fila en Neon (o la lógica en el servidor); no hace falta republicar la app solo por eso.
  • Seguridad: la cadena completa de Neon puede vivir solo en el servidor; la app no necesita la "llave" de administrador de la base.
  • Multiplataforma: el mismo Backend puede atender Android, web u otros clientes en el futuro.

Resumen: el Backend es la oficina central; la API es la ventanilla por donde el Frontend pide y recibe información. La base Neon es el almacén donde persisten pedidos, usuarios, versiones, etc.

5. Pasos secuenciales (si armás algo parecido de nuevo)

  1. Definir el modelo de datos (pedidos, usuarios, roles, estados) y escribir el SQL inicial para PostgreSQL.
  2. Crear el proyecto en Neon, ejecutar los scripts y guardar la cadena de conexión solo en secretos o .env (nunca en repo público).
  3. Implementar una API mínima (Node + Express + cliente pg): rutas necesarias, CORS acotado, JWT si hay usuarios.
  4. Desplegar la API (por ejemplo en Render) y configurar variables de entorno.
  5. Desarrollar el front web (mapa con Leaflet, formularios, listas) y conectarlo a la API o al flujo de datos que definas.
  6. Publicar la web en GitHub con Pages y, si corresponde, Actions para inyectar configuración sin secretos en el repo.
  7. Crear la app Android con WebView que cargue el index desde assets o la URL de Pages; puentes JavaScript para cámara, archivos, sesión y notificaciones.
  8. Configurar actualizaciones OTA: tabla app_version en Neon, endpoint en la API, archivo app_update_config.json en el APK y APK alojado con URL directa.
  9. Probar en cadena: web local → web en Pages → APK debug → APK release firmado → subir APK → actualizar fila en Neon → probar “buscar actualización”.
  10. Documentar en docs/ y revisar que no se suban credenciales ni keystores.

6. Checklist de una página (operativa)

Rutas típicas respecto del repositorio Nexxo (Android). En el repo de Pages, la paridad suele ser index.html y sw.js en la raíz.

6.1. Modelo y base de datos

  • Definir tablas y roles.
  • Scripts SQL en docs/ (por ejemplo docs/NEON_app_version.sql y otros docs/NEON_*.sql).
  • Ejecutar en Neon SQL Editor; connection string solo en secretos / .env.

6.2. API backend

  • Código en api/ (server.js, package.json).
  • Variables de entorno en el host (Render u otro).
  • Probar en local: npm install y node server.js (o npm run dev).
  • Desplegar y verificar rutas (por ejemplo /api/app-version para OTA).

6.3. Front web

  • Fuente principal: app/src/main/assets/index.html.
  • Service worker: app/src/main/assets/sw.js.
  • Plantilla de config: app/src/main/assets/config.example.json (no subir config.json con secretos).
  • Probar en navegador.

6.4. Repositorio web (GitHub Pages)

  • Sincronizar index.html y sw.js con Nexxo.
  • config.example.json y workflow en .github/workflows/ si usás Actions.
  • Publicar y probar la URL de Pages.

6.5. App Android

  • Versión en app/build.gradle.kts (versionCode / versionName).
  • Assets: index.html, sw.js.
  • OTA: app/src/main/assets/app_update_config.json apuntando a la API.
  • Firma: keystore.properties local.
  • Build assembleDebug / assembleRelease y prueba en dispositivo.

6.6. Distribución del APK

  • Subir .apk a Drive u otro servicio con enlace de descarga directa.
  • Actualizar Neon (INSERT/UPDATE en app_version); referencia en docs/NEON_app_version.sql.
  • Comprobar que la app detecta la nueva versión.

6.7. Documentación y seguridad

  • Revisar docs/.
  • No commitear: config.json real, keystores, cadenas de conexión.
  • README con enlace a Pages y pasos de despliegue.

6.8. Cierre

  • Prueba integral: web + Android + API + Neon.
  • Commit y push en Nexxo y, si aplica, sync al repo de Pages.
